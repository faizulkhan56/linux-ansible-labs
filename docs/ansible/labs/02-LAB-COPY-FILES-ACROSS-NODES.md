# Lab 02b — Copy files to all nodes (`copy` module)

## Objective

- Push files from the **control node** to **every managed host** using the same inventory patterns as earlier labs (`nodes`, `all`, etc.).
- Practice **ad-hoc** `copy` and a small **playbook** with the usual options: permissions, ownership, backups, directories, and inline `content`.
- Know when **`fetch`** (pull from host) differs from **`copy`** (push to host).

Prerequisite: Lab 01 (inventory) and Lab 02 (ad-hoc basics). Next in sequence: Lab 03 (packages and services).

## Step 0 — Setup

```bash
mkdir -p ~/ansible-labs/lab02-copy/files
cd ~/ansible-labs/lab02-copy
```

Copy your inventory from Lab 01:

```bash
cp ~/ansible-labs/lab01/inventory.ini .
```

Use a host pattern that matches your inventory (`nodes`, `all`, `masters`, `slaves`, etc.). Examples below use `nodes`; substitute if needed.

Create a sample file on the **control node** only (Ansible will upload it):

```bash
echo "lab-managed-banner v1" > files/banner.txt
```

## Step 1 — Ad-hoc: copy one file to the same path on all nodes

```bash
ansible -i inventory.ini nodes -b -m ansible.builtin.copy -a "src=files/banner.txt dest=/tmp/lab-banner.txt mode=0644"
```

Theory — `copy` vs shell `cp`

- `ansible.builtin.copy` reads `src` from the **controller** (paths are relative to the playbook directory in playbooks; for ad-hoc, use a path relative to your current directory or an absolute path).
- It is **idempotent**: if the file already matches (content and relevant attributes), Ansible reports `ok` instead of `changed`.
- Prefer `copy` over `shell: cp` so checks, permissions, and reporting stay consistent.

Command breakdown

- `-b` (become): needed if `dest` is not writable by the SSH user (here `/tmp` is usually fine without `-b`; keep `-b` for consistency with system paths later).
- `src=files/banner.txt`: local file on the control node.
- `dest=/tmp/lab-banner.txt`: path on each managed node.
- `mode=0644`: file mode on the remote (octal string or symbolic form).

Verify on all nodes:

```bash
ansible -i inventory.ini nodes -m ansible.builtin.command -a "cat /tmp/lab-banner.txt"
```

## Step 2 — Playbook: copy with owner, group, and backup

Create `copy-lab.yml`:

```yaml
- name: Distribute a managed file with ownership and backup
  hosts: nodes
  become: true
  tasks:
    - name: Ensure lab drop-in directory exists
      ansible.builtin.file:
        path: /etc/lab-dropin
        state: directory
        mode: "0755"

    - name: Install banner from controller with owner and backup on change
      ansible.builtin.copy:
        src: files/banner.txt
        dest: /etc/lab-dropin/banner.txt
        owner: root
        group: root
        mode: "0644"
        backup: true
```

Run:

```bash
ansible-playbook -i inventory.ini copy-lab.yml
```

Run again: second run should show **`ok`** (no change) if nothing drifted.

Theory — `backup: true`

- Before overwriting a changed file, Ansible saves a timestamped copy next to the target (e.g. `banner.txt.12345.2025-01-01@12:00:00~`).

## Step 3 — Inline file with `content` (no `src` file)

Useful for tiny files or generated one-liners.

Ad-hoc example:

```bash
ansible -i inventory.ini nodes -b -m ansible.builtin.copy -a "content='DO NOT EDIT ON HOST\nGenerated by Ansible lab\n' dest=/etc/lab-dropin/README mode=0644 owner=root group=root"
```

Playbook task equivalent:

```yaml
    - name: Drop a small README from inline content
      ansible.builtin.copy:
        dest: /etc/lab-dropin/README
        content: |
          DO NOT EDIT ON HOST
          Generated by Ansible lab
        owner: root
        group: root
        mode: "0644"
```

Note: use **`content`** *or* **`src`**, not both.

## Step 4 — Copy a whole directory (recursive)

Place several files under `files/motd.d/` on the controller:

```bash
mkdir -p files/motd.d
echo "Welcome from Ansible" > files/motd.d/10-ansible-welcome
echo "Contact: lab@example" > files/motd.d/20-contact
```

Task:

```yaml
    - name: Sync a directory of snippets to remote
      ansible.builtin.copy:
        src: files/motd.d/
        dest: /etc/lab-dropin/motd.d/
        owner: root
        group: root
        mode: preserve
```

Theory — Trailing slash on `src`

- `src: files/motd.d/` (with trailing slash): copy **contents** into `dest` (like `rsync` trailing-slash behavior).
- `src: files/motd.d` (no slash): copy the **directory** as a single child under `dest`.

Pick the layout you want under `/etc/lab-dropin/motd.d/` and test with `ls -la` on one host.

## Step 5 — `remote_src`: copy from the remote disk (not from controller)

`copy` normally uploads from the controller. With `remote_src: true`, `src` is a path **on the managed node** (copy/rename on that host only).

Example (ad-hoc):

```bash
ansible -i inventory.ini nodes -b -m ansible.builtin.copy -a "src=/etc/os-release dest=/tmp/os-release-copy.txt remote_src=true mode=0644"
```

Use cases: staging a file already downloaded on the host, or duplicating a known path without pulling it to the controller first.

## Step 6 — Optional: `fetch` (pull one file from each node to the controller)

Opposite direction: each host’s file is copied **to** the controller. By default Ansible creates a **per-host** tree under `dest` (safe for multiple nodes).

```bash
ansible -i inventory.ini nodes -m ansible.builtin.fetch -a "src=/etc/os-release dest=/tmp/ansible-fetch-lab"
```

On the controller, list what arrived (paths include the hostname and the original path):

```bash
find /tmp/ansible-fetch-lab -type f
```

- `flat=yes` places a single file directly under `dest` without hostname subdirs; use it only when the play targets **one** host, otherwise filenames collide.

## Step 7 — Cleanup (optional)

```bash
ansible -i inventory.ini nodes -b -m ansible.builtin.file -a "path=/etc/lab-dropin state=absent"
ansible -i inventory.ini nodes -b -m ansible.builtin.file -a "path=/tmp/lab-banner.txt state=absent"
ansible -i inventory.ini nodes -b -m ansible.builtin.file -a "path=/tmp/os-release-copy.txt state=absent"
```

## Reference — Common `copy` options

| Option | Purpose |
|--------|---------|
| `src` / `content` | File on controller or inline string |
| `dest` | Remote path (file or directory parent, depending on task) |
| `owner`, `group`, `mode` | Metadata (`become` usually required outside user-writable dirs) |
| `backup` | Keep a dated backup before replace |
| `force` | `no` can refuse to replace certain read-only scenarios; default is to replace when content differs |
| `remote_src` | Treat `src` as path on managed node |
| `validate` | Run a command with `%s` placeholder before committing (e.g. `visudo -cf %s`) |
| `decrypt` | With vault-encrypted `src` (see Lab 07) |

## What this lab did not cover (by design)

- **`synchronize`** (`rsync`): great for huge trees; often needs `delegate_to` or SSH quirks; see [ansible.posix.synchronize](https://docs.ansible.com/ansible/latest/collections/ansible/posix/synchronize_module.html) when `copy` is too slow.
- **`template`**: files with variables (`{{ }}`); see Lab 04.

## Notes

- If you see “Could not find or access `files/...` on the Ansible Controller”, run the playbook from the directory that contains `files/`, or use an absolute `src`, or place files under a role’s `files/` layout.
- If the host pattern `nodes` is undefined, use `all` or add `[nodes:children]` to your inventory (Lab 01 / Lab 02 examples).
